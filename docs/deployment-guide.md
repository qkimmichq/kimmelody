# Kimmelody 部署指南 — 从原理到实践

## 前言：什么叫"部署"？

你现在开发时，是在自己电脑上跑项目——打开终端，`npm run dev`，然后浏览器访问 `localhost:8080`。

部署的本质问题是：**怎么让这个项目在"另一台机器"上跑起来，并且能稳定地一直跑？**

"另一台机器"可能是：
- 你家里的一台旧电脑（7x24 开机当服务器）
- 一台云服务器（阿里云/腾讯云租的虚拟机）
- 别人的手机（做成 App）

下面按从简单到复杂的顺序，逐一解释每种方案在解决什么问题、原理是什么。

---

## 一、直接运行 vs 进程守护

### 1.1 开发模式的问题

你现在启动项目的方式：

```bash
# 终端 1
cd packages/music-api && node app.js -p 3000

# 终端 2
npm run dev
```

这种方式有三个问题：

1. **终端关了，进程就死了**——进程是挂在终端会话上的，关终端 = 杀进程
2. **崩溃了不会自动重启**——代码报错 → 进程退出 → 服务挂了，需要手动重开
3. **没有日志管理**——所有输出刷在终端里，关了终端就丢了

### 1.2 PM2 解决什么问题

PM2 本质上是一个**进程保姆**。你把进程"托管"给它，它负责：

| 能力 | 原理 |
|------|------|
| 后台运行（daemon） | PM2 自身是一个后台守护进程，它启动的子进程不依附于任何终端 |
| 崩溃自动重启 | PM2 监控子进程的退出信号，发现异常退出就重新 `fork` 一个 |
| 日志持久化 | 把 stdout/stderr 重定向到磁盘文件，自动按大小切分 |
| 开机自启 | 往系统的 init 系统（systemd/launchd）注册一条启动命令 |

**本质**：PM2 = 一个专门管 Node.js 进程的 supervisor。它不改变你的代码，只是在外面套了一层保护。

### 1.3 为什么还不够

PM2 解决了"进程不挂"，但没解决另一个问题：**环境一致性**。

你的电脑上有 Node 20、装好了所有 npm 依赖、配置好了 `.env`。换一台机器呢？Node 版本对不对？系统依赖装没装？这些都需要人工操作，容易遗漏。

这就引出了 Docker。

## 二、Docker：让环境"可复制"

### 2.1 Docker 是什么

一句话：**Docker 把你的项目 + 运行环境打包成一个标准化的"箱子"，这个箱子在任何机器上都能原样运行。**

类比：你搬家时把东西装进纸箱——箱子里的物品布局不变，不管箱子搬到哪间屋子。Docker 就是这个纸箱，里面装的是你的代码 + Node.js + npm 依赖 + 系统工具。

### 2.2 三个核心概念

```
Dockerfile  ──(build)──►  Image  ──(run)──►  Container
   (配方)                   (模板)              (运行实例)
```

- **Dockerfile**：一个文本文件，描述"怎么构建这个环境"。比如：从 Node 20 镜像开始 → 复制代码进去 → `npm install` → 设置启动命令。
- **Image（镜像）**：Dockerfile 构建出来的只读模板。类似 `.iso` 安装光盘。
- **Container（容器）**：从镜像启动的运行实例。类似从光盘安装后跑起来的操作系统。容器之间互相隔离。

### 2.3 为什么是"隔离"的

每个容器有自己独立的：
- **文件系统**：容器内的 `/app/data` 和宿主机的 `/data` 是两回事，不显式映射就互不可见
- **网络**：默认每个容器有自己的虚拟网卡和 IP
- **进程空间**：容器 A 里的进程看不到容器 B 里的进程

所以 Docker 不是虚拟机——它共享宿主机的 Linux 内核，但通过 Linux 的 namespace/cgroups 机制做资源隔离。启动快（秒级），占用少（几十 MB 内存）。

### 2.4 Docker Compose：编排多个容器

你的项目需要两个进程：Kimmelody Server（8080）+ NeteaseCloudMusicApi（3000）。

Docker Compose 用一个 YAML 文件描述"这俩容器怎么一起跑"：

```
docker-compose.yml  ──(定义)──►  ┌─ Container: server (8080)
                                  │     └─ 网络连接 ──┐
                                  └─ Container: music-api (3000) ←┘
```

关键细节：两个容器之间用**内部 Docker 网络**通信。`music-api` 容器不对外暴露端口（只有 `server` 容器能访问它），安全性更好。

Compose 还处理了：
- 启动顺序（`depends_on`：先启 music-api，再启 server）
- 数据持久化（`volumes`：把容器内的 `/app/data` 映射到宿主机目录，容器删了数据还在）
- 统一启停（`docker compose up/down` 一条命令操作所有服务）

### 2.5 Docker 方案的好处

| 场景 | 无 Docker | 有 Docker |
|------|----------|-----------|
| 换一台新电脑 | 装 Node → `npm install` → 配 `.env` → 手动开两个终端 | `docker compose up` 一行 |
| 项目依赖升级 | 可能和系统其他项目冲突 | 容器内独立，互不影响 |
| 部署到服务器 | 同样的手动流程 + 可能系统差异 | 完全一致，本地能跑服务器就能跑 |
| 给别人用 | 对方得装 Node、配环境 | 对方只需装 Docker |

## 三、PWA：把网页"变成"App

### 3.1 原理

PWA（Progressive Web App）的核心思想：**用浏览器提供的原生能力，让网页看起来/用起来像一个真正的 App。**

你的项目已经在 HTML 里写了两行关键代码：

```html
<meta name="apple-mobile-web-app-capable" content="yes">  <!-- iOS：允许全屏 -->
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">
```

这两行的效果：当用户在 iOS Safari 里点"添加到主屏幕"后，打开的网页没有浏览器地址栏、没有底部工具栏——全屏显示，跟原生 App 一样。

### 3.2 两个关键部件

**manifest.json** — 告诉浏览器"我是一个 App"：

浏览器读取这个文件，知道：
- 这个应用叫什么（`name`）
- 用什么图标（`icons`）
- 打开时要不要隐藏浏览器 UI（`display: standalone`）
- 主题色是什么（顶栏颜色）

**Service Worker** — 离线缓存引擎：

这是一个独立的 JS 文件，运行在浏览器后台（不是网页线程），可以：
- 拦截所有网络请求，决定从缓存拿还是从网络拿
- 在你首次打开时预缓存静态资源（CSS/JS/图片）
- 这样即使没有网络，App 也能打开（离线可用）

它不依赖服务器配合，纯前端能力。

### 3.3 PWA 能做什么、不能做什么

| 能做到 | 做不到 |
|--------|--------|
| 手机桌面有图标，像 App | 上架 App Store / Google Play |
| 全屏无浏览器 UI | 调用蓝牙、NFC 等底层硬件 |
| 离线打开（缓存静态资源） | 无网络时使用需要后端的功能（搜索、AI） |
| 推送通知（Chrome/Edge 支持） | iOS 推送通知（Safari 不支持，需用原生） |
| 自动更新（检测新版本） | 后台长时间运行 |

### 3.4 适用场景

个人项目 / 小范围使用 / 不想折腾应用商店审核 → PWA 是性价比最高的"App 化"方案。你的后端功能不变，前端只是在浏览器上多了一层包装。

## 四、Tauri / Electron：真正的桌面 App

### 4.1 它们是什么

PWA 的局限是你必须通过浏览器打开。Tauri 和 Electron 把这一步也省了——**它们自带一个浏览器引擎，打包进你的安装包。**

| 方案 | 内置浏览器引擎 | 安装包大小 | 内存占用 |
|------|-------------|----------|---------|
| Electron | Chromium（完整 Chrome 内核） | ~150 MB | ~200 MB |
| Tauri | 系统自带 WebView（Windows 用 Edge WebView2，macOS 用 WKWebView） | ~5 MB | ~20 MB |

### 4.2 Tauri 的原理

Tauri 由两部分组成：

```
Tauri App
├── Rust 后端（主进程）
│   ├── 创建窗口
│   ├── 调用系统 API（文件系统、通知、托盘图标）
│   └── 如果你的后端是 Node.js → 通过 Sidecar 机制启动 Node 进程
└── WebView（渲染你的前端 HTML/CSS/JS）
```

关键点：**Tauri 不包含后端逻辑。**它只负责开一个原生窗口 + 加载你的前端页面。你的 Express 服务仍然需要独立运行——Tauri 可以通过 Sidecar 功能在你打开 App 时自动启动它。

### 4.3 和 PWA 的区别

| | PWA | Tauri |
|---|-----|-------|
| 分发方式 | 网址 / 浏览器安装 | .exe / .dmg / .apk 文件 |
| 用户体验 | 接近原生 | 完全原生窗口 |
| 系统能力 | 有限（取决于浏览器） | 完整（通过 Rust 调用） |
| 开发成本 | 极低 | 中等（需配置 Rust 工具链） |
| 更新 | 自动（刷新即更新） | 需推送新版本安装包 |

## 五、云端部署：让全世界访问

### 5.1 为什么需要云服务器

目前你的项目跑在 `localhost`，只有你自己的电脑能访问。

要让别人也能用，你需要一台**有公网 IP、24 小时不关机的电脑**。这就是云服务器（VPS，Virtual Private Server）。

本质上它就是机房里的一个虚拟机，给你 root 权限，你可以装任何软件。和你本地电脑一样，只不过：
- 有固定公网 IP
- 7x24 运行
- 你通过 SSH 远程登录操作（命令行，没有桌面）

### 5.2 为什么不能"无服务器"（Serverless）

有一种流行的部署方式叫 Serverless（如 Vercel、Netlify、AWS Lambda）：你把代码传上去，平台自动分配资源跑你的函数，按调用次数付费。

但你的项目不适合 Serverless，原因：

**WebSocket 长连接**：你的项目用 WebSocket 做实时聊天 + 状态同步。Serverless 平台的设计模式是"来一个请求 → 启一个函数 → 返回 → 销毁函数"。WebSocket 需要持续的 TCP 连接，不能被切成一个个独立的函数调用。

**音频代理流式转发**：你为了绕过网易云 CORS，做了 `/audio/proxy` 端点，它把 MP3 文件流式转发给前端。Serverless 按请求次数 + 出流量计费，一首 5MB 的歌可能触发几毛钱费用，听一天歌就破产了。

**Scheduler 定时任务**：你的 DJ 自动调度依赖 `setInterval` 在进程内跑。Serverless 函数跑完就销毁了，没有"持续运行的进程"这个概念。

**结论**：你的架构需要的是一个**长驻进程**，而不是按需调用的函数。所以必须用传统 VPS。

### 5.3 部署链路的每一环

```
用户手机
   │
   ▼  互联网
┌──────────────┐
│  Nginx       │  ← 门户：SSL 加密 + 路由分发
│  (反向代理)   │
└──────┬───────┘
       │
       ▼  localhost:8080
┌──────────────┐
│  Express     │  ← 你的后端
│  (Kimmelody) │
└──────┬───────┘
       │  localhost:3000
       ▼
┌──────────────┐
│  Music API   │  ← 网易云代理
└──────────────┘
```

**Nginx 是什么？**一个高性能的 Web 服务器，常做"反向代理"：

- 用户访问 `https://你的域名` → Nginx 接收请求 → 转发给 `localhost:8080`（你的 Express）
- 这就叫"反向代理"——用户的请求不直接打到你的 Node 进程，Nginx 站在前面做中间人
- Nginx 还处理 SSL 证书（HTTPS 加密）、静态文件加速、限流、防攻击

**为什么不能直接用 Express 对外？**可以，但 Nginx 更擅长这些事：
- HTTPS 性能更好
- 可以同时托管多个服务（比如未来加一个博客）
- 更安全（请求先经过 Nginx，可以过滤恶意流量）

### 5.4 Cloudflare Tunnel：没有公网 IP 也能用

普通的家庭宽带（电信/联通/移动）给的是**运营商内网 IP**，外网访问不到你的电脑。

Cloudflare Tunnel 的原理：

```
你的电脑                       Cloudflare 服务器            用户
┌──────────┐    加密隧道      ┌──────────────┐     HTTPS     ┌─────┐
│ localhost │ ◄────────────► │ 公网 IP+域名   │ ◄─────────── │ 手机 │
│  :8080    │   (你主动发起)   │ 你的域名.com   │              └─────┘
└──────────┘                 └──────────────┘
```

你在本地装一个 `cloudflared` 程序，它主动去连 Cloudflare 的服务器并建立一条加密隧道。外部用户访问你的域名 → 请求走到 Cloudflare → 通过隧道转发到你本地。

**关键**：是你主动连出去的，不需要防火墙开端口，也不需要公网 IP。

**适用场景**：
- 在家里的旧电脑/NAS 上跑 Kimmelody
- 没有公网 IP
- 不想买云服务器
- 带宽取决于你的家庭宽带上传速度

**成本**：Cloudflare Tunnel 免费，但需要一个域名（`.com` 约 ¥50/年，`.xyz` 等便宜域名几块钱一年）。

## 七、建议的渐进路线

你现在还在开发阶段，功能会频繁变动。建议按这个顺序逐步推进，每一步价值独立，不需要一步到位：

1. **PM2**（现在就能做）—— 不用每次手动开两个终端，`pm2 start` 后就能一直挂着。不影响开发，只是省掉重复操作。

2. **Docker Compose**（功能稳定后再做）—— 把运行环境固化下来，换电脑也能一键跑。后续部署到 VPS 也是用同一套 Docker 配置。

3. **PWA 补齐**（前端基本定稿后做）—— 加个 Service Worker + manifest 图标，手机就能"安装"了。前端每次部署自动更新。

4. **VPS 部署**（想给别人用时再做）—— 买一台轻量云，Docker 配置直接搬过去，Nginx + SSL 一配就上线。

前三步和功能开发不冲突，可以并行推进。
